DailyCut 웹 프론트엔드 상세 구현 명세서
1. 사이트 목적 및 대상 사용자
사이트 목적:

DailyCut은 사용자가 현재 확보한 '가용 시간(분 단위)'과 '구독 중인 OTT 플랫폼' 정보를 교차 분석하여, 남은 시간 안에 끝까지 시청할 수 있는 최적의 영상 콘텐츠를 즉각적으로 추천해 주는 웹 플랫폼이다.
수많은 콘텐츠 속에서 무엇을 볼지 고르느라 오히려 시간을 낭비하는 '넷플릭스 증후군(결정 장애)'을 해결하는 것을 최우선 목표로 한다.

이 사이트의 핵심 목적은 다음과 같다.

사용자의 남은 시간에 정확히 들어맞는(Time-Fit) 콘텐츠를 필터링하여 제공한다.

사용자가 결제하여 이용 중인 OTT 플랫폼 내에서만 볼 수 있는 현실적인 결과를 제공한다.

타이핑을 최소화하고 직관적인 터치(클릭)만으로 3초 안에 추천을 받을 수 있는 극강의 편의성을 제공한다.

단순한 나열이 아니라, "왜 지금 이 콘텐츠를 봐야 하는지(러닝타임 비교 등)"를 명확한 UI로 설득한다.

향후 AI 기반 개인화 추천, 장르 필터링, 글로벌 다국어 지원이 붙어도 무너지지 않는 컴포넌트 구조를 설계한다.

대상 사용자

주요 대상 사용자는 다음과 같다.

식사 시간(30분~1시간) 동안 짧게 볼 이른바 '밥친구(영상)'가 필요한 1인 가구 및 직장인

출퇴근, 통학 등 대중교통 이동 시간이 규칙적으로 발생하는 사람

약속 시간 전 애매하게 남은 시간(예: 40분)을 때워야 하는 사람

수많은 OTT 콘텐츠 목록을 스크롤만 하다가 지쳐버리는 헤비 구독자

사용자 니즈

이 사이트는 다음 니즈를 해결해야 한다.

콘텐츠의 러닝타임을 일일이 들어가서 확인하기 귀찮은 문제

내가 구독하지 않은 플랫폼의 독점작이 추천되어 느끼는 박탈감 문제

시간은 없는데 텍스트로 시간을 직접 타이핑해서 입력해야 하는 번거로움

따라서 사이트는 직관적이고 빠른 퀵 버튼 UI, 시각적으로 압도적인 포스터 기반 결과 화면, 가용 시간과 러닝타임의 직관적인 뱃지 비교, 모바일 환경(특히 한 손 조작)에 최적화된 사용성을 핵심 가치로 가져야 한다.

2. 주요 기능 및 페이지 구성
필수 페이지

홈 (Landing & Input Page)

첫 방문자가 DailyCut의 핵심 가치를 직관적으로 이해하고, 즉시 탐색을 시작할 수 있는 진입 페이지이다.

포함 요소:

Hero Section: 강한 공감을 이끌어내는 카피와 서비스 정체성 시각화

Time Input (시간 입력 폼):

직접 숫자 입력 필드 (모바일 숫자 키패드 최적화)

Quick Time Buttons: [+30분], [+45분], [+1시간], [+2시간] 등 원클릭으로 시간을 누적 또는 설정하는 버튼

OTT Selector (플랫폼 선택 UI):

Netflix, Disney+, TVING, Wavve, Watcha, Coupang Play 등의 로고 기반 토글 버튼

다중 선택 가능, 선택 시 브랜드 컬러 테두리 활성화

CTA 버튼:

"내 시간에 딱 맞는 영상 찾기"

시간이나 OTT가 선택되지 않았을 때는 비활성화(Disabled) 처리 및 안내 툴팁 제공

로딩 화면 (Loading State / Overlay)

API 데이터 호출 및 필터링 과정에서 발생할 수 있는 대기 시간(1~3초) 동안 사용자의 이탈을 막는 화면이다.

포함 요소:

시계 바늘이 빠르게 돌아가거나 팝콘이 튀겨지는 Lottie 또는 CSS 애니메이션

동적 로딩 문구 (사용자 입력값 반영)

예: "입력하신 45분에 딱 맞는 넷플릭스 명작을 찾는 중..."

추천 결과 페이지 (Result Page)

백엔드의 Time-Fit 알고리즘을 거쳐 도출된 최적의 콘텐츠 목록을 시각적으로 화려하게 보여주는 핵심 페이지이다.

포함 요소:

결과 요약 헤더:

"⏱ 45분 이내 • 넷플릭스, 티빙" 형태의 요약 칩(Chip)

"조건 다시 설정하기" 버튼

추천 콘텐츠 카드 리스트 (Grid Layout):

각 카드에 포함될 정보:

TMDB 고해상도 포스터 이미지 (레이지 로딩 적용)

콘텐츠 제목 (최대 2줄, 말줄임 처리)

Time Badge (핵심): "러닝타임 42분 (3분 여유!)" 형태의 강조 뱃지

시청 가능한 OTT 플랫폼 로고 아이콘 배열

Empty State (검색 결과 없음):

조건에 맞는 콘텐츠가 없을 경우 표시

예: "앗, 해당 시간 안에 볼 수 있는 영상이 없어요. 시간을 10분만 늘려보시겠어요?"

시간 연장 퀵 버튼 제공

404 페이지 (Not Found Page)

존재하지 않는 경로로 접근했을 때 보여주는 페이지이다.

포함 요소:

위트 있는 오류 메시지 (예: "시간이 멈춘 공간입니다. 404")

홈으로 돌아가기 버튼

3. 주요 기능 요구사항
공통 기능

모바일 최우선(Mobile-First) 반응형 웹 디자인 적용 (스마트폰 사용 비율이 90% 이상일 것으로 예상)

영상 콘텐츠가 돋보이고 어두운 환경에서 눈이 편안한 Dark Mode(다크 테마) 전용 기반 UI 구현

상태 관리(Zustand 또는 Context API)를 통해 홈 화면에서 입력한 데이터를 결과 화면으로 안전하게 전달

URL Query Parameters(예: ?time=45&otts=netflix)를 활용하여 결과 페이지 링크 공유 가능하도록 설계

디자인 방향

DailyCut의 디자인은 다음 키워드를 반영할 것.

Cinematic: 영화관 같은 몰입감 (딥 블랙 배경)

Instant: 즉각적이고 군더더기 없는 입력 폼

Intuitive: 글을 읽지 않아도 아이콘과 색상만으로 이해되는 직관성

Neon Accent: 넷플릭스 레드 등 각 OTT의 고유 컬러를 포인트 컬러로 활용

디자인 스타일 가이드:

배경색: #141414 (디스플레이의 진한 검정) 또는 #0F172A (Slate 계열 어두운 색)

타이포그래피: 숫자가 돋보이는 폰트(Pretendard 등), 굵기와 크기 대비를 확실히 주어 가독성 확보

카드 UI 적극 활용하되, 테두리(border)보다는 은은한 그림자(drop-shadow)와 배경색 차이로 구분

애니메이션 / 인터랙션

퀵 버튼 및 OTT 토글 클릭 시 가벼운 scale-95 (눌림) 효과

결과 카드 호버 시 살짝 떠오르며(-translate-y-1) 정보가 밝아지는 효과

페이지 전환 시 부드러운 페이드 인/아웃

SEO 및 성능

포스터 이미지는 반드시 TMDB의 최적화된 사이즈(w500)를 사용

IntersectionObserver를 활용한 이미지 레이지 로딩(Lazy Loading) 적용

semantic HTML 사용 (main, section, article 등)

4. 콘텐츠 요구사항
홈 페이지 주요 문구 예시

메인 헤드라인:
고르다 끝나는 휴식 시간, 이제 그만.

서브 문구:
지금 가진 여유 시간과 구독 중인 OTT를 알려주세요. 고민 없이 바로 시작할 수 있는 콘텐츠를 찾아드립니다.

입력 라벨:
"지금 얼마 동안 볼 수 있나요?" / "어떤 플랫폼을 구독 중인가요?"

결과 페이지 문구 예시

결과 타이틀:
완벽한 타이밍을 위한 DailyCut 추천작

카드 내 뱃지 텍스트:
"딱 맞아요!", "5분 여유", "조금 촉박함" 등 러닝타임 대비 여유 시간에 따른 동적 텍스트

이미지 및 아이콘

OTT 로고는 투명 배경의 SVG 포맷 사용 필수

아이콘은 lucide-react 라이브러리의 얇고 모던한 라인 아이콘 사용 (타이머, 모래시계, 화살표 등)

포스터 이미지가 없는 콘텐츠를 대비한 로고 워터마크가 박힌 Fallback(대체) 이미지 준비

5. 성능 및 접근성 요구사항
성능 최적화

렌더링 최적화: React.memo를 활용하여 OTT 토글 버튼 등 잦은 상태 변경과 무관한 컴포넌트의 불필요한 리렌더링 방지

외부 API 호출 최소화: 백엔드에서 데이터를 받아올 때 한 번에 필요한 메타데이터(포스터 경로, 시간 등)를 모두 가져오도록 협의

CLS(Cumulative Layout Shift) 방지: 이미지가 로드되기 전, 카드의 고정된 높이 비율(예: 2:3)을 유지하는 Skeleton UI 적용

접근성(a11y)

OTT 선택 토글은 시각 장애인 리더기를 위해 aria-pressed 상태값과 숨김 텍스트(예: "넷플릭스 선택됨") 제공

포스터 이미지에는 반드시 alt="[영화제목] 포스터" 속성 부여

폼 입력은 마우스 없이 키보드(Tab, Enter)만으로도 전부 조작 가능하도록 tabIndex 관리

7. 프레임워크 및 기술 스택
이 프로젝트는 다음 환경을 기준으로 작업해야 한다.

Core: React 18, TypeScript, Vite

Routing: React Router DOM v6

Styling: Tailwind CSS (빠른 반응형 대응 및 컴포넌트 수준의 스타일링)

State Management: Zustand (또는 React Context API - 단일 플로우이므로 무거운 Redux는 지양)

Data Fetching: Axios (추후 여력이 된다면 데이터 캐싱을 위해 TanStack React Query 도입 고려)

Icons: Lucide-React

구조 원칙

기능(Feature) 단위의 디렉토리 분리를 원칙으로 한다.


추후 다국어나 관리자 기능이 붙을 수 있음을 고려하여 하드코딩된 텍스트는 가급적 상수로 관리한다.

9. 코드 스타일 및 작업 방식
작업 단위: 한 번에 전체를 만들지 않고 컴포넌트 단위로 Bottom-Up 방식으로 작업한다.

컴포넌트 분리: HomePage.tsx 하나에 모든 코드를 넣지 않고, TimeInput, OttSelector 등 의미 있는 덩어리로 반드시 분리한다.

타입스크립트 적용: 모든 Props와 API 응답 데이터는 types/ 폴더 내에 인터페이스(Interface)로 정의하여 any 타입 사용을 엄격히 금지한다.

목업 우선: 백엔드 API가 완성되지 않았더라도, 구조화된 Mock 데이터를 먼저 만들어 프론트 UI 개발을 블로킹 없이 진행한다.

불필요한 파일 삭제 지양: 기본 세팅된 구조를 파괴하지 않고 추가/수정하는 방식으로 진행한다.

10. 1차 구현 목표 (MVP)
1차 목표는 동작하는 프론트 UI 골격(Mock 데이터 기반) 완성이다.

1차 구현 범위:

다크 모드 기반 메인 레이아웃 및 헤더

홈 화면의 시간 입력 로직 (직접 입력 및 퀵 버튼 상태 연동)

홈 화면의 OTT 선택 로직 (다중 배열 상태 관리)

폼 제출 시 유효성 검사 후 /result?time=X&otts=Y,Z 로 라우팅 이동

결과 화면에서 URL 파라미터를 읽어 Mock 데이터 리스트를 필터링

ContentCard 컴포넌트를 활용한 반응형 그리드 렌더링

현재 단계에서는 백엔드 연동 없이도 프론트엔드 단독으로 완벽하게 동작하고 시연 가능한 "클릭 가능한 프로토타입(Clickable Prototype)" 수준을 완성하는 데 집중한다.
